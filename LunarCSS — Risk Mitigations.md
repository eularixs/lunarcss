# LunarCSS — Risk Mitigations

**Version:** 1.0.0
**Author:** Eularix Team

***

## Risk 1 — Babel Transform + Reanimated Coexist

**Problem:**
Using a Babel plugin for the `className → __lcssTw()` transform runs in the same Babel pipeline as `react-native-reanimated/plugin`. Plugin order becomes fragile — this is the exact same root cause as NativeWind v4's conflict issues. Worklet boundaries can be corrupted if the wrong plugin runs first.

**Solution: Replace Babel plugin with Metro Transformer**

The transform is moved to the Metro layer, which runs **before** Babel entirely. Reanimated's Babel plugin never sees LunarCSS-modified AST and worklet boundaries are never touched.

```ts
// lunarcss/metro/transformer.ts
import { transform as upstreamTransform } from 'metro-transform-plugins'

export function transform({ src, filename, options }) {
  // Step 1: LunarCSS transforms className → __lcssTw() at file source level
  const transformed = transformClassNames(src, filename)

  // Step 2: Pass to upstream Babel transformer unchanged
  // Reanimated plugin runs here — completely isolated from Step 1
  return upstreamTransform({ src: transformed, filename, options })
}
```

```js
// metro.config.js — generated by lunar init
const { getDefaultConfig } = require('expo/metro-config')
const { withLunarCSS } = require('lunarcss/metro')

const config = getDefaultConfig(__dirname)
module.exports = withLunarCSS(config)
```

**Result:**

- Zero plugin order dependency
- Worklet boundaries fully intact
- No `jsxImportSource` — Reanimated never conflicts
- Expo Go compatible — no JSI, pure JS transformer

## Risk 2 — OKLCH → sRGB Lossy Conversion

**Problem:**
Tailwind v4 defaults to OKLCH color space. React Native only supports hex and rgb. Converting wide-gamut P3 OKLCH values to sRGB causes color clamping — visible color shift on wide-gamut displays (iPhone 12+, modern Android).

**Solution: Two-path color handling + dev warning**

- **Web** — OKLCH is passed as-is via CSS custom properties. The browser renders P3 natively. No conversion, no loss.
- **Mobile** — `culori` converts OKLCH to nearest sRGB hex. Clamping is inevitable for out-of-gamut values and is documented behavior.

```ts
// resolver/v4/utilities/colors.ts
import { oklch, formatHex, clampChroma } from 'culori'

export function resolveOklch(value: string): string {
  const color = oklch(value)
  const clamped = clampChroma(color, 'oklch') // clamp to sRGB gamut
  const hex = formatHex(clamped)

  if (isDev && isOutOfSRGBGamut(color)) {
    console.warn(
      `[LunarCSS] Color "${value}" is outside sRGB gamut and has been clamped to "${hex}" on mobile. ` +
      `Use an sRGB-safe OKLCH value or provide a hex fallback.`
    )
  }

  return hex
}
```

**Documented behavior for users:**

```css
@theme {
  /* Wide gamut — will clamp on mobile, full P3 on web */
  --color-primary: oklch(0.6 0.2 264);

  /* sRGB-safe alternative */
  --color-primary: oklch(0.6 0.15 264);

  /* Or provide hex directly for mobile precision */
  --color-primary: #6366f1;
}
```

## Risk 3 — Build-Time Static Extraction Cache Key Strategy

**Problem:**
Static extraction only catches literal strings. Template literals and conditional expressions fall back to runtime. Cache key collisions or stale entries can occur if the same class string resolves differently under different theme states.

**Solution: Composite cache key = className + themeHash**

```ts
// runtime/cache.ts
let currentThemeHash = computeThemeHash(defaultTokens)

const styleCache = new Map<string, ReturnType<typeof StyleSheet.create>>()

export function resolveTw(className: string) {
  const cacheKey = `${className}::${currentThemeHash}`

  if (styleCache.has(cacheKey)) return styleCache.get(cacheKey)!

  const resolved = StyleSheet.create({ s: parseAndResolve(className) }).s
  styleCache.set(cacheKey, resolved)
  return resolved
}
```

**Static extraction behavior:**

- Literal strings → extracted at build-time, baked into `LCSS_STATIC` — O(1) lookup, zero runtime compute
- Template literals (`bg-${color}-500`) → skip extraction, always resolved at runtime via cache
- Conditional expressions (`isActive ? 'bg-primary' : 'bg-zinc-100'`) → both branches extracted independently at build-time if both are static strings

```ts
// Build-time extracted output
export const LCSS_STATIC = StyleSheet.create({
  "text-sm font-bold": { fontSize: 14, fontWeight: '700' },
  "bg-primary px-4": { backgroundColor: '#6366f1', paddingHorizontal: 16 },
  // both branches of conditionals extracted
  "bg-primary": { backgroundColor: '#6366f1' },
  "bg-zinc-100": { backgroundColor: '#f4f4f5' },
})
```

## Risk 4 — Dynamic Theming + Static Extract Conflict

**Problem:**
Pre-extracted styles bake token values at build-time. Calling `updateTheme()` at runtime does not update already-extracted `LCSS_STATIC` styles — leading to stale styles for components using changed tokens.

**Solution: Two-tier system — static for default theme, runtime for dynamic**

Static extraction is treated as the **initial/default theme only**. Any call to `updateTheme()` bypasses static styles entirely and uses the runtime resolver.

```ts
// runtime/theme.ts
import { themeEmitter } from './emitter'
import { styleCache, invalidateThemeHash } from './cache'
import { getClassesUsingTokens } from './resolver'

export function updateTheme(tokens: Record<string, string>) {
  // 1. Update token registry
  applyTokens(tokens)

  // 2. Invalidate runtime cache entries using changed tokens
  const affectedClasses = getClassesUsingTokens(Object.keys(tokens))
  affectedClasses.forEach(key => styleCache.delete(key))

  // 3. Recompute themeHash — all subsequent cache lookups use new key
  invalidateThemeHash()

  // 4. Notify all subscribed components to re-render
  themeEmitter.emit('change')
}
```

**Component subscription — zero Provider required (Opsi B):**

```ts
// runtime/useTheme.ts
export function useTheme() {
  const [, forceUpdate] = useReducer(x => x + 1, 0)

  useEffect(() => {
    themeEmitter.on('change', forceUpdate)
    return () => themeEmitter.off('change', forceUpdate)
  }, [])
}

// Auto-subscribed in __lcssTw() — user does not need to call this manually
```

**Documented behavior:**
> Static extraction applies to the default theme only. Styles resolved after a `updateTheme()` call always go through the runtime resolver and are cached under the new theme hash.

## Risk 5 — Modifier Resolution Order

**Problem:**
Stacked modifiers like `ios:dark:active:bg-primary` have no defined precedence rules. Ambiguous order leads to unpredictable output.

**Solution: Explicit four-tier precedence chain**

```
Tier 1 (Platform)    → ios: / android: / web:
Tier 2 (ColorScheme) → dark: / light:
Tier 3 (State)       → active: / disabled: / focus: / pressed:
Tier 4 (Responsive)  → sm: / md: / lg: / xl:
```

Rules:

- Modifiers are processed **left to right**, later same-tier modifiers win
- A modifier chain is only applied if **all conditions in the chain are true**
- Conflicting same-tier modifiers (e.g. `ios:android:`) → last one wins + dev warning

```ts
// runtime/platform.ts
export function resolveModifiers(className: string): ViewStyle | null {
  const { modifiers, base } = parseModifiers(className)
  // e.g. "ios:dark:active:bg-primary" → modifiers: ['ios', 'dark', 'active'], base: 'bg-primary'

  // Tier 1 — Platform
  if (modifiers.includes('ios') && Platform.OS !== 'ios') return null
  if (modifiers.includes('android') && Platform.OS !== 'android') return null
  if (modifiers.includes('web') && Platform.OS !== 'web') return null

  // Tier 2 — ColorScheme
  if (modifiers.includes('dark') && colorScheme !== 'dark') return null
  if (modifiers.includes('light') && colorScheme !== 'light') return null

  // Tier 3 — State (injected via component props)
  if (modifiers.includes('active') && !isActive) return null
  if (modifiers.includes('disabled') && !isDisabled) return null

  // Tier 4 — Responsive
  if (modifiers.includes('sm') && windowWidth >= breakpoints.sm) return null

  // All conditions passed — resolve base class
  return resolveTw(base)
}
```

## Risk 6 — Next.js Turbopack Alias Fragility

**Problem:**
`react-native → react-native-web` alias injection via `next.config.ts` is fragile across Next.js versions. Turbopack config API changed between Next.js 14, 15, and 16.

**Solution: Version-aware injection + post-init verification**

```ts
// cli/detect.ts
export function injectNextConfig(configPath: string) {
  const nextVersion = getNextVersion()

  if (nextVersion >= 16) {
    // Turbopack default — inject turbopack + transpilePackages
    injectTurbopackAlias(configPath)
  } else if (nextVersion >= 13) {
    // Webpack — inject webpack alias + transpilePackages
    injectWebpackAlias(configPath)
  } else {
    throw new Error('[LunarCSS] Next.js 13+ required')
  }
}
```

```ts
// Generated next.config.ts for Next.js 16+
const nextConfig: NextConfig = {
  transpilePackages: ['react-native', 'react-native-web', 'lunarcss'],
  turbopack: {
    resolveAlias: { 'react-native': 'react-native-web' },
    resolveExtensions: ['.web.tsx', '.web.ts', '.web.js', '.tsx', '.ts', '.js']
  }
}
```

**Post-init verification:**

```ts
// lunar init runs a dry-run import check after injection
async function verifyAlias() {
  try {
    await resolveImport('react-native', { from: process.cwd() })
    console.log('✓ react-native alias verified')
  } catch {
    console.warn(
      '[LunarCSS] Alias verification failed. Manual fix may be required.\n' +
      'See: https://lunarcss.dev/docs/nextjs-setup'
    )
  }
}
```

## Risk 7 — Dual v3/v4 Detection + pnpm Hoisting

**Problem:**
`require.resolve('tailwindcss/package.json')` fails under pnpm due to non-flat `node_modules` hoisting. Version detection silently falls back to wrong resolver.

**Solution: Multi-strategy detection with explicit fallback chain**

```ts
// resolver/detect.ts
export function getTailwindVersion(): number {
  // Strategy 1: Standard resolve from project root (npm / yarn / bun)
  try {
    const pkgPath = require.resolve('tailwindcss/package.json', {
      paths: [process.cwd()]
    })
    return parseInt(require(pkgPath).version.split('.')[0])
  } catch {}

  // Strategy 2: pnpm — walk node_modules/.pnpm directory
  try {
    return detectFromPnpmStore()
  } catch {}

  // Strategy 3: Parse lockfile (pnpm-lock.yaml / yarn.lock / bun.lockb)
  try {
    return detectFromLockfile()
  } catch {}

  // Strategy 4: Check tailwind config file presence
  // tailwind.config.js = v3, no config file = likely v4
  try {
    if (existsSync(path.join(process.cwd(), 'tailwind.config.js'))) return 3
    if (existsSync(path.join(process.cwd(), 'tailwind.config.ts'))) return 3
    return 4
  } catch {}

  // Final fallback: warn + default to v4
  console.warn(
    '[LunarCSS] Could not detect Tailwind version. Defaulting to v4. ' +
    'Set version manually in lunar.config.ts: { tailwindVersion: 3 }'
  )
  return 4
}
```

**Manual override escape hatch:**

```ts
// lunar.config.ts
export default defineConfig({
  tailwindVersion: 3, // force v3 resolver if auto-detection fails
})
```

## Risk 8 — Metro Transformer Source Map Preservation

**Problem:**
String-level rewrite of `className → __lcssTw()` shifts column/line offsets. Source maps drift, breaking stack traces, breakpoints, and error overlays in Expo/Metro. Debugging UX degrades silently.

**Solution: AST-based transform with offset-preserving codegen**

Use `@babel/parser` + `@babel/generator` with `retainLines: true` and `sourceMaps: true`, OR `magic-string` for surgical edits that preserve original offsets. Never use raw regex string replace on user source.

```ts
// metro/transformer.ts
import { parse } from '@babel/parser'
import generate from '@babel/generator'
import traverse from '@babel/traverse'

export function transformClassNames(src: string, filename: string) {
  const ast = parse(src, { sourceType: 'module', plugins: ['jsx', 'typescript'] })

  traverse(ast, {
    JSXAttribute(path) {
      if (path.node.name.name !== 'className') return
      // rewrite to style={__lcssTw(...)}
    },
  })

  const { code, map } = generate(ast, {
    retainLines: true,
    sourceMaps: true,
    sourceFileName: filename,
  })

  return { code, map }
}
```

**Test:** trigger runtime error in transformed component, verify stack trace points to original `className` line, not transformed line.

## Risk 9 — Build-Time Extraction Whole-File Analysis Limits

**Problem:**
Naive literal detection on `<View className="..." />` misses common patterns:

- `<View {...props} className="..." />` — spread before className
- `<MyButton className="px-4" />` — custom wrappers (not intrinsic RN element)
- `cn("px-4", isActive && "bg-primary")` — helper composition
- `clsx`/`classnames` library usage

Result: most real-world strings fall to runtime path, defeating extraction goal.

**Solution: Scoped extraction + explicit wrapper allowlist + helper recognition**

```ts
// babel-plugin/build-extractor.ts
const RN_INTRINSICS = new Set(['View', 'Text', 'Pressable', 'ScrollView', 'Image', 'TextInput'])
const KNOWN_HELPERS = new Set(['cn', 'clsx', 'classnames', 'tw'])

// Extract when:
// 1. JSXAttribute name === 'className'
// 2. Value is StringLiteral OR ConditionalExpression with both branches StringLiteral
// 3. Element is intrinsic RN OR in user-configured wrapper allowlist
// 4. OR call expression to known helper with literal string args
```

**User config escape hatch:**

```ts
// lunar.config.ts
export default defineConfig({
  extract: {
    components: ['MyButton', 'Card', 'Box'], // wrapper allowlist
    helpers: ['cn', 'twMerge'],              // helper allowlist
  }
})
```

Document: spreads + dynamic helpers fall to runtime cache. Not a bug, by design.

## Risk 10 — `__lcssTw` Runtime in Web Bundle

**Problem:**
Even with web passthrough, shared components (used both native + web) get Babel-transformed once. `__lcssTw` import lands in web bundle, adding dead weight. Tree-shaking does not remove because runtime call is reachable.

**Solution: Platform-specific entry via `.web.ts` resolver extension + no-op stub**

```ts
// runtime/tw.ts          → mobile resolver (full)
// runtime/tw.web.ts      → no-op passthrough that returns className string

// runtime/tw.web.ts
export function __lcssTw(className: string): { className: string } {
  return { className } // RN-Web reads className, ignores style on DOM elements
}
```

Metro/Webpack/Turbopack auto-pick `.web.ts` for web target. Runtime stub <200 bytes.

**Alternative for Next.js DOM elements:** Babel transformer detects `<div>`/`<span>`/`<p>` and skips transform entirely (per PRD §5.1).

## Risk 11 — `updateTheme()` Linear Cache Walk

**Problem:**
`getClassesUsingTokens()` over full `styleCache` is O(n). Lunar Colony low-code builders fire `updateTheme()` on every slider drag → 60fps × O(n) = jank.

**Solution: Reverse index `token → Set<className>` built incrementally**

```ts
// runtime/cache.ts
const tokenIndex = new Map<string, Set<string>>() // tokenName → set of cacheKeys

function recordTokenUsage(cacheKey: string, tokensUsed: string[]) {
  for (const t of tokensUsed) {
    if (!tokenIndex.has(t)) tokenIndex.set(t, new Set())
    tokenIndex.get(t)!.add(cacheKey)
  }
}

export function invalidateTokens(tokens: string[]) {
  for (const t of tokens) {
    const keys = tokenIndex.get(t)
    if (!keys) continue
    for (const k of keys) styleCache.delete(k)
    tokenIndex.delete(t)
  }
}
```

`parseAndResolve()` returns `{ style, tokensUsed }`. Cache write also writes to index. Invalidation now O(changedTokens × affectedClasses), not O(totalCache).

## Risk 12 — Metro Cache Stale on `lunar.css` Edit

**Problem:**
Metro caches transform output by source file hash. User edits `@theme` tokens in `lunar.css` → dependent JS files unchanged → Metro reuses cached extracted styles → stale colors until full cache wipe.

**Solution: Declare `lunar.css` as transform dependency + content-hash invalidation**

```ts
// metro/transformer.ts
export function transform({ src, filename, options }) {
  const themeFile = findLunarCss(options.projectRoot)
  const themeHash = hashFile(themeFile) // sha1 of lunar.css contents

  const transformed = transformClassNames(src, filename, { themeHash })

  return {
    ...upstreamTransform({ src: transformed, filename, options }),
    dependencies: [
      ...existingDeps,
      { name: themeFile, data: { key: themeHash, isAsync: false } },
    ],
  }
}
```

Metro graph re-transforms files when declared dependency hash changes. Editing `lunar.css` invalidates all dependents automatically.

**Fallback:** `lunar` CLI emits `lunar.css.lock` next to `lunar.css` containing parsed tokens. Watch mode reads lock for hot updates.

## Risk 13 — TypeScript `className` Prop Missing on RN Components

**Problem:**
`react-native` types do not declare `className` on `ViewProps`/`TextProps`/etc. User code `<View className="px-4" />` → TS error 2322. Cannot ship without ambient augmentation.

**Solution: Module augmentation shipped as `lunarcss/types` + auto-include via `lunar init`**

```ts
// types/react-native.d.ts (shipped with package)
import 'react-native'

declare module 'react-native' {
  interface ViewProps { className?: string }
  interface TextProps { className?: string }
  interface ImageProps { className?: string }
  interface ScrollViewProps { className?: string }
  interface PressableProps { className?: string }
  interface TextInputProps { className?: string }
  interface SectionListProps<ItemT, SectionT> { className?: string }
  interface FlatListProps<ItemT> { className?: string }
}
```

`lunar init` injects into `tsconfig.json`:

```json
{ "compilerOptions": { "types": ["lunarcss/types"] } }
```

Or user adds manual reference: `/// <reference types="lunarcss/types" />` in entry file.

Document: Tier-1 RN components covered. Third-party libraries (Reanimated `<Animated.View>`, gesture-handler) need user-side augmentation — provide copy-paste snippet in docs.

## Risk 14 — Unbounded Runtime Cache Growth

**Problem:**
Template literal classes (`bg-${color}-${shade} p-${size}`) generate combinatorial cache entries. App with 256 colors × 11 shades × 8 sizes = 22,528 entries. Memory leak on long-lived screens.

**Solution: LRU cap on runtime cache (NOT static extraction)**

```ts
// runtime/cache.ts
import { LRUCache } from 'lru-cache' // or hand-rolled, ~50 lines

const styleCache = new LRUCache<string, ReturnType<typeof StyleSheet.create>>({
  max: 1000, // configurable via lunar.config.ts
})
```

Static extraction stays unbounded (build-time bounded by source code size). Runtime cap only applies to dynamic path.

**User-tunable:**

```ts
// lunar.config.ts
export default defineConfig({
  runtime: { cacheSize: 2000 }
})
```

Hand-rolled LRU preferred to avoid `lru-cache` dep (~6kb). Bundle target tight.

## Risk 15 — `@theme` Parser Bundle Size

**Problem:**
Full PostCSS parser is ~200kb minified — blows full-package <20kb gzip target. Cannot ship as resolver dependency.

**Solution: Minimal hand-rolled `@theme` block parser**

Subset supported:

- `@theme { --token: value; }` — flat key/value pairs
- Comments `/* */` stripped
- `@import "lunarcss";` recognized but no-op at parse layer
- Reject: nested rules, `@media`, `@supports`, selectors, calc beyond simple

```ts
// resolver/v4/parser.ts
const THEME_BLOCK = /@theme\s*\{([^}]*)\}/g

export function parseTheme(css: string): Record<string, string> {
  const tokens: Record<string, string> = {}
  const stripped = stripComments(css)
  const match = THEME_BLOCK.exec(stripped) // RegExp.prototype.exec — not child_process
  if (!match) return tokens

  for (const decl of match[1].split(';')) {
    const [key, ...rest] = decl.split(':')
    if (!key || !rest.length) continue
    const k = key.trim()
    const v = rest.join(':').trim()
    if (k.startsWith('--')) tokens[k] = v
  }
  return tokens
}
```

Target: <2kb. Document subset clearly. Reject unsupported syntax with line number.

**Future:** if user demand emerges for nested/conditional themes, ship optional `lunarcss/parser-full` dynamic import.

## Risk 16 — RN `StyleSheet.create` Reject Unsupported v4 Props

**Problem:**
Tailwind v4 introduces: `@container` queries, 3D transforms (`rotate-x-*`, `perspective-*`), `color-mix()`, `@starting-style`, view transitions. RN `StyleSheet` validation either silently drops unknown keys (release) or throws (dev).

**Solution: Per-utility-group platform-capability whitelist + skip-with-warn**

```ts
// resolver/v4/utilities/transforms.ts
const RN_SUPPORTED = new Set([
  'transform', 'rotate', 'scale', 'translateX', 'translateY',
  // 3D not supported — rotateX/Y/Z work via transform array but perspective limited
])

export function resolveTransform(className: string): ViewStyle | null {
  const props = parseTransformClass(className)

  for (const key of Object.keys(props)) {
    if (!RN_SUPPORTED.has(key)) {
      if (isDev) console.warn(`[LunarCSS] "${className}" uses "${key}" not supported on RN. Skipped.`)
      delete props[key]
    }
  }
  return Object.keys(props).length ? props : null
}
```

**Compatibility matrix in docs** per utility group:

| Utility | Web | iOS | Android | Notes |
|---|---|---|---|---|
| `@container` | ✅ | ❌ | ❌ | Use `useWindowDimensions` instead |
| `rotate-x-*` 3D | ✅ | ⚠️ | ⚠️ | Limited via transform matrix |
| `color-mix()` | ✅ | ❌ | ❌ | Pre-compute with culori in @theme |
| `bg-[url(...)]` | ✅ | ❌ | ❌ | Use `<Image source>` |

## Risk 18 — `lunar init` Idempotency

**Problem:**
Re-running `lunar init` (CI re-runs, dev re-runs after corruption) must not double-inject babel plugin, duplicate `@plugin "lunarcss"` line, or repeat `transpilePackages` entries. Regex append is fragile.

**Solution: AST-based merge with presence checks**

For each target file:

- `babel.config.js` — parse AST, check `plugins` array for existing `'lunarcss/babel-plugin'` string OR identifier reference. Skip if present.
- `metro.config.js` — check for `withLunarCSS(` call expression. Skip if present.
- `next.config.ts` — parse object literal, merge into `transpilePackages` / `turbopack.resolveAlias` only if key/value missing.
- `global.css` — line scan for `@plugin "lunarcss"`. Skip if present.
- `tsconfig.json` — JSON parse, merge `compilerOptions.types` array uniquely.

```ts
// cli/init.ts
async function injectIdempotent(file: string, mutator: (ast: Node) => boolean) {
  const ast = parseFile(file)
  const changed = mutator(ast) // returns true if anything actually added
  if (!changed) {
    console.log(`✓ ${file} already configured, skipped`)
    return
  }
  await writeFile(file, generate(ast).code)
}
```

**Dry-run mode:** `lunar init --dry-run` prints proposed diffs without writing. Useful for CI assertions.
